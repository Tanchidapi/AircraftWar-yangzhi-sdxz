## 一、项目概述（10 分）

> 简要描述项目内容，并说明实施意义。

**项目内容：** 本项目是以《软件构造》课程中开发的 PC 端"飞机大战"Java 程序为基础，利用 Android Studio 开发环境，将其迁移至移动端。开发过程涉及 UI 适配、触屏交互重写、Android 异步机制应用、音效系统集成，并基于网络功能实现全球排行榜与用户数据同步。

**实施意义：**

- **架构深度理解：** 从 Java Swing 的桌面开发跨越到 Android 五层系统架构，掌握 Activity 生命周期与移动端资源管理。
- **代码复用与重构实践：** 通过迁移具有多种设计模式（单例、工厂、策略等）的复杂项目，提升在不同平台间重用核心业务逻辑的能力。
- **功能演进：** 使传统的单机 PC 游戏具备移动互联属性，学习 Android 异步请求处理与服务器交互。

## 二、原有功能迁移分析（30 分）

> 描述选择迁移的代码、待迁移的 Windows 端代码中可复用的部分及需重新适配安卓平台的功能模块，并分别说明原因。

### 1. 选择迁移哪位小组成员的代码，为什么？

选择迁移小组成员**糜宇航**的代码。

**原因：** 该代码具有极高的软件构造规范性。代码结构中严格应用了单例模式（HeroCanvas/HeroAircraft）、工厂模式（不同敌机的产生）以及模板方法模式（BaseGame 定义游戏骨架）。其模块化程度高，业务逻辑与 UI 显示分离清晰，极大降低了向 Android 平台迁移时的逻辑重写成本。

### 2. 可复用模块及依据

- **核心飞行对象模型（Domain Model）：** 包括 AbstractFlyingObject 及其子类（英雄机、精英机、子弹等）。**依据：** Android 完全支持 Java/Kotlin 语法，这些纯逻辑类不依赖 GUI 库，可以直接拷贝使用。
- **设计模式架构：** 原有代码中的工厂模式（产生敌机和道具）和策略/模板模式（不同难度模式的处理）。**依据：** 这些模式属于软件构造的逻辑层，在 Android 的数据处理和逻辑控制中完全通用。
- **碰撞检测算法：** 如判断矩形相交的数学逻辑。**依据：** 物理逻辑与平台无关。

### 3. 需重新开发的模块及依据

- **图形渲染引擎：** 需将基于 javax.swing 和 Graphics 的绘制逻辑替换为 Android 的 SurfaceView 或 Canvas 绘图系统。**依据：** 移动端采用硬件加速和不同的刷新机制（如双缓冲在 Android 中的实现方式不同）。
- **用户交互监听：** 需将 KeyListener（键盘操作）重写为 OnTouchListener（触屏手势和多点触控）。**依据：** 移动设备缺失物理键盘，需适配点击、滑动等交互。
- **多线程与异步控制：** 需将原有的简单 Thread 运行逻辑适配为 Android 的 Handler 消息机制或 ExecutorService。**依据：** Android 严禁在主线程进行耗时操作或网络请求。

## 三、新增网络功能设计（30 分）

> 至少规划 2 项网络功能，分别说明功能内容、设计原因及简要技术实现思路。

### 功能一：在线排行榜（HTTP/OkHttp 实现）

- **设计原因：** 增强玩家间的竞争性，解决单机版无法保存全球排名的问题。
- **技术思路：** 利用 OkHttp 向云端 API 发送 POST 请求提交分数，使用 GET 请求获取 JSON 格式的排名列表，并利用 Handler 或 LiveData 更新 UI。

### 功能二：实时对战/多人协助（Socket 实现）

- **设计原因：** 提升游戏互动性，实现多名玩家在同一局内竞技。
- **技术思路：** 建立 Socket 长连接，实时同步不同客户端的玩家坐标和子弹信息，采用多线程处理网络 IO 以保证游戏流畅度。
